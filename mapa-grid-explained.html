01

Модуль и правило FIT

Базовый кирпич — прозрачный квадрат фиксированного размера (по умолчанию 300×300). Внутрь помещается фото по правилу FIT: вписать целиком, сохранив пропорции. Поскольку пропорции фото и квадрата обычно не совпадают, с одной стороны остаётся пустое поле — и правило говорит, к какой границе фото прижать.

Решение принимается по двум входным данным: ориентация фото (считается из его реальных размеров) и номер колонки, в которой стоит модуль.

ar = ширина_фото / высота_фото // пропорция // ЕСЛИ фото горизонтальное или квадратное (ar ≥ 1) фото по ширине модуля, прижато к низу // ЕСЛИ фото вертикальное (ar < 1) и колонка 1–2 фото по высоте модуля, прижато влево // ЕСЛИ фото вертикальное (ar < 1) и колонка 3–4 фото по высоте модуля, прижато вправо
горизонт / квадратFIT по низу
вертикаль · кол. 1–2FIT влево
вертикаль · кол. 3–4FIT вправо

Пунктир — границы прозрачного модуля, заливка — фото. Красная риска показывает, откуда стартует подпись — об этом следующий слой.

02

Текст от левого края фото

Подпись под модулем верстается не от края модуля, а от левого края фото внутри него. Для горизонтального фото и для вертикали слева это одно и то же (левый край фото совпадает с модулем). А вот для вертикали, прижатой вправо, фото сдвинуто — и подпись уезжает вместе с ним.

Сдвиг считается из пропорции, а не измеряется по факту (это важно: измерять DOM до раскладки ненадёжно):

// ширина вписанного вертикального фото при высоте модуля mod ширина_фото = mod · ar // левый край фото = отступ подписи photoLeft = mod − mod · ar // для FIT вправо photoLeft = 0 // для низа и FIT влево

Урок отладки. Первая версия считала этот отступ через clientWidth модуля — и он выходил нулевым, потому что встроенные картинки (data-URI) готовы синхронно, и расчёт срабатывал до первой раскладки, когда размеры ещё нули. Лечится расчётом от константы mod и пропорции фото.

03

Сетка: шахматка, смещение, просвет

Модули расставлены по 4 колонкам в шахматном порядке. Это задаёт диагональный ритм:

  • Нечётные строки (1, 3, 5…) — модули в колонках 1 и 3.
  • Чётные строки (2, 4, 6…) — модули в колонках 2 и 4.
  • Внутри строки второй модуль опущен на смещение относительно первого — строка идёт по диагонали.

Вертикальная раскладка держится на трёх числах: размер модуля mod, смещение внутри строки offset, просвет между строками gap. Из них складывается шаг строки:

rs = mod + offset + gap // шаг между строками верхняя_линия_строки(row) = BASE + row · rs верх модуля колонок 1,2 = верхняя_линия верх модуля колонок 3,4 = верхняя_линия + offset

04

Адаптивность и поля экрана

Размер модуля фиксирован, а вот горизонтальные позиции колонок считаются от ширины окна. Крайняя левая колонка стоит у левого поля, крайняя правая упирается правым краём в правое поле — по 20px с каждой стороны. Всё между ними делится поровну:

step = (ширина_экрана − 2·20 − mod) / 3 // шаг между колонками x(колонка) = 20 + (колонка − 1) · step

Когда экрана не хватает, step становится меньше ширины модуля — и колонки начинают наезжать друг на друга. Это не ошибка, а способ уложить сетку в любой экран: по вертикали строки разведены, поэтому наезжающие по горизонтали модули складываются в плотную диагональную ленту.

Две грабли, которые ловились замером, а не на глаз. Дефолтный <figure> несёт браузерный margin: 0 40px — он молча толкал всю сетку вправо, пока не сбросили. А ширину надо брать у видимой области (documentElement.clientWidth): если считать по элементу, вертикальный скроллбар «съедает» правое поле, и оно выходит меньше левого.

05

Параллакс между строками

Идея: внизу экрана строки расставлены широко, к верху сближаются. При скролле строка входит снизу «растянутой» и, поднимаясь, сжимается — нижняя визуально догоняет верхнюю. Расстояние между строками перестаёт быть постоянным и становится функцией положения строки на экране.

Сдвиг каждой строки — гладкая функция её экранной позиции t (0 — верх экрана, 1 — низ):

t = (позиция_строки − скролл) / высота_экрана amp = сила_параллакса · высота_экрана · 0.3 dy = amp · ( cos(2π·t) − 1 ) / 2 // сдвиг строки

Почему именно косинус. Ранняя версия брала степенную кривую только внутри экрана, а снаружи — линию. Значения на кромке совпадали, но скорость строки скакала — и каждая строка, пересекая край экрана при скролле, дёргалась. У косинуса производная на обеих кромках равна нулю, поэтому строка входит и выходит из зоны эффекта плавно. Причину дёрганья, кстати, нашли замером: сначала грешили на скачущую высоту документа — но scrollHeight оказался стабильным, и это отсекло ложную версию.

06

Перенос на WebGL

Пока сетка жила на DOM, параллакс двигал элементы через CSS-трансформации на каждый кадр скролла — это нагружало главный поток и перерисовывало тяжёлые фото. На WebGL всё это уходит на видеокарту, и рендер становится дешёвым.

  • Каждое фото загружается как текстура, модуль — это прямоугольник (quad) с этой текстурой.
  • FIT считается геометрией quad'а: по пропорции текстуры вычисляются размер и положение фото внутри модуля.
  • Подписи рисуются в отдельные текстуры (текст → 2D-канвас → текстура) и ставятся от левого края фото.
  • Скролл — нативный: невидимый блок держит высоту документа, а полотно на весь экран рисует то, что попадает во вьюпорт.
  • Фон — цвет очистки кадра; цвет подписей автоматически светлеет на тёмном фоне.

Упрощённый цикл отрисовки одного кадра:

на каждый кадр (скролл / изменение параметра): очистить кадр цветом фона для каждой видимой строки: экранная_Y = верхняя_линия − скролл + dy // параллакс для каждого модуля строки: x = позиция колонки (px, py, pw, ph) = FIT(текстура, колонка) // вписать фото нарисовать quad с текстурой фото нарисовать quad с текстурой подписи от px

Тонкость. Рендер вызывается по событию (скролл, изменение параметра), а не в бесконечном цикле — экономнее. Но тогда WebGL-контекст нужно создавать с preserveDrawingBuffer: true, иначе между кадрами буфер очищается и полотно моргает пустотой.

Модель собрана итеративно: сначала одно правило FIT, затем сетка, адаптив, параллакс и перенос на WebGL. Каждый спорный шаг проверялся замером в реальном рендере, а не на глаз — это и есть главный приём вайб-кодинга здесь.
